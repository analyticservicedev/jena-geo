Detailed Description

A set of static methods working on images.

Some of those methods are useful, but not really rigorous. This is why they do not appear in any "official" package, but instead in this private one. 

                 <strong>Do not rely on this API!</strong>

It may change in incompatible way in any future version.

static Dimension toTileSize ( final Dimension  size)
static

Suggests a tile size for the specified image size.

On input,

size

is the image size. On output, it is the tile size. This method write the result directly in the supplied object and returns

size

for convenience.

This method it aimed to computing a tile size such that the tile grid would have overlapped the image bound in order to avoid having tiles crossing the image bounds and being therefore partially empty. This method will never returns a tile size smaller than { MIN_TILE_SIZE}. If this method can't suggest a size, then it left the corresponding

size

field (width or height) unchanged.

The width and height fields are processed independently in the same way. The following discussion use the

width

field as an example.

This method inspects different tile sizes close to the JAI::getDefaultTileSize() default tile size}. Let

width

be the default tile width. Values are tried in the following order:

width

,

width+1

,

width-1

,

width+2

,

width-2

,

width+3

,

width-3

, etc. until one of the following happen:

  • A suitable tile size is found. More specifically, a size is found which is a divisor of the specified image size, and is the closest one of the default tile size. The Dimension field (

    width

    or

    height

    ) is set to this value.

  • An arbitrary limit (both a minimum and a maximum tile size) is reached. In this case, this method may set the Dimension field to a value that maximize the remainder of image size / tile size (in other words, the size that left as few empty pixels as possible).

273  int sopt = 0; // An "optimal" tile size, to be used if no exact dividor is found.
274  int rmax = 0; // The remainder of 'imageSize / sopt'. We will try to maximize this value.
275  /*
276  * Inspects all tile sizes in the range [GEOTOOLKIT_MIN_TILE_SIZE .. MAX_TIME_SIZE]. We will begin
277  * with a tile size equal to the specified 'tileSize'. Next we will try tile sizes of
278  * 'tileSize+1', 'tileSize-1', 'tileSize+2', 'tileSize-2', 'tileSize+3', 'tileSize-3',
279  * etc. until a tile size if found suitable.
280  *
281  * More generally, the loop below tests the 'tileSize+i' and 'tileSize-i' values. The
282  * 'stop' constant was computed assuming that MIN_TIME_SIZE < tileSize < MAX_TILE_SIZE.
283  * If a tile size is found which is a dividor of the image size, than that tile size (the
284  * closest one to 'tileSize') is returned. Otherwise, the loop continue until all values
285  * in the range [GEOTOOLKIT_MIN_TILE_SIZE .. MAX_TIME_SIZE] were tested. In this process, we
286  * remind the tile size that gave the greatest reminder (rmax). In other words, this is the
287  * tile size with the smallest amount of empty pixels.
288  */
289  for (int i=0; i<=stop; i++) {
290  int s;
291  if ((s = tileSize+i) <= MAX_TILE_SIZE) {
292  final int r = imageSize % s;
293  if (r == 0) {
294  // Found a size >= to 'tileSize' which is a dividor of image size.
295  return s;
296  }
297  if (r > rmax) {
298  rmax = r;
299  sopt = s;
300  }
301  }
302  if ((s = tileSize-i) >= MIN_TILE_SIZE) {
303  final int r = imageSize % s;
304  if (r == 0) {
305  // Found a size <= to 'tileSize' which is a dividor of image size.
306  return s;
307  }
308  if (r > rmax) {
309  rmax = r;
310  sopt = s;
311  }
312  }
313  }
314  /*
315  * No dividor were found in the range [GEOTOOLKIT_MIN_TILE_SIZE .. MAX_TIME_SIZE]. At this point
316  * 'sopt' is an "optimal" tile size (the one that left as few empty pixel as possible),
317  * and 'rmax' is the amount of non-empty pixels using this tile size. We will use this
318  * "optimal" tile size only if it fill at least 75% of the tile. Otherwise, we arbitrarily
319  * consider that it doesn't worth to use a "non-standard" tile size. The purpose of this
320  * arbitrary test is again to avoid too many small tiles (assuming that
321  */
322  return (rmax >= tileSize - tileSize/4) ? sopt : 0;
323  }

◆ STRIPE_SIZE

final int STRIPE_SIZE = 64
staticprivate

Maximum tile width or height before to consider a tile as a stripe.

It tile width or height are smaller or equals than this size, then the image will be re-tiled. That is done because there are many formats that use stripes as an alternative to tiles, an example is TIFF. A stripe can be a performance issue, users can have stripes as large as 20000 columns x 8 rows. If we just want to see a chunk of 512x512, this is a lot of unneeded data to load.
